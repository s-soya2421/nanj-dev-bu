# ADR Policy

議論の末に設計判断が固まった場合、`docs/adr/` に昇格させる。

## ADR を作るべきケース

- 技術選定を決めた
- データモデルを決めた
- インフラ構成の判断を決めた
- 将来的な制約を伴う判断をした
- 「なぜそれを採用したか」を残す価値がある

## ADR を作らなくてよいケース

- 自明な実装判断
- 影響範囲がそのファイル内に閉じている判断
- 後から容易に変更できる判断

## ADR の書き方ルール

- **ADR は遊ばない**。標準日本語で簡潔に書く
- 状況、決定、理由、影響を書く
- 関連する `history/threads/` と `specs/` をリンクする
- 一度 Accepted になった ADR は原則書き換えず、撤回する場合は新しい ADR を起票して古い方を Superseded にする

## Status の値

- `Proposed` — 提案中
- `Accepted` — 採用
- `Superseded by ADR-XXXX` — より新しい ADR に置き換えられた
- `Deprecated` — 廃止
