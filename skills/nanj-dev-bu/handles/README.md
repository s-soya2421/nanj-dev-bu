# ハンドル選びの指針

## 基本ルール

- 1スレッドあたり **3〜5 人** の固定参加者を選ぶ
- 同じハンドルは、そのスレッド内で**専門性・立場・口調・問題意識**を維持する
- 途中で都合よく別人格化しない
- 前の発言と矛盾しない
- 単発ネタではなく、議論が進むようコメントする
- 同じハンドルは、過去コメントを踏まえて補足・修正・掘り下げを行う

## 命名思想

- IT / Web / インフラ / AI / 開発文化 に寄せた、名無し文化風の名前
- 名前を見るだけで、その人物の専門や役回りが少し分かる状態が望ましい
- スレごとにランダムにするより、役割ベースで一貫させる

### 命名パターン

- `名無しさん＠○○`
- `名無し○○民`
- `風吹けば名無し○○`
- `名無し○○警察`
- `名無し○○職人`

## カタログ

| ファイル | ハンドル | 役割 |
|---|---|---|
| `pm.md` | 風吹けば名無しPM | MVP・優先順位・スコープ整理 |
| `type-safety.md` | 名無しさん＠型安全 | 型・責務分離・境界 |
| `sql-police.md` | 名無しSQL警察 | DB設計・整合性・性能 |
| `sre.md` | 風吹けば名無しSRE | 運用・監視・障害対応 |
| `gpu.md` | 名無しGPU民 | AI・推論コスト・将来拡張 |
| `app-router.md` | 名無しAppRouter | Next.js / RSC・URL駆動 |
| `observability.md` | 名無しさん＠可観測性 | ログ設計・追跡可能性 |
| `dto-craftsman.md` | 名無しDTO職人 | API境界・整形責務 |
| `scope-boss.md` | 名無しスコープ番長 | 話が広がるとMVPに戻す |

## 選び方の目安

- **機能設計スレ** — PM + 型安全 + SQL警察 + SRE
- **技術選定スレ** — 型安全 + AppRouter + DTO職人 + スコープ番長
- **データモデルスレ** — SQL警察 + 型安全 + PM
- **AI機能スレ** — GPU民 + PM + SRE + 可観測性
- **運用方針スレ** — SRE + 可観測性 + PM
